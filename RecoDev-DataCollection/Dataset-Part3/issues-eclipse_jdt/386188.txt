This came from the fix to bug 380243, where we try telling the parser to ignore loading external DTDs for validation. See http://xerces.apache.org/xerces-j/features.html for more information about parser features.

I guess in the case that the JRE does not support the used feature, we should just ignore the feature failure and try to continue parsing.
pushed fix to: http://git.eclipse.org/c/jdt/eclipse.jdt.debug.git/commit/?id=16e4336fd9b54f3a5cef8fa0e1ac453cc3adae58

to just continue parsing with the default features.

We could also look at over-riding the entity resolver (as mentioned on the interwebs), but that hack feels like it would slow down an already slow operation even further (loading / validating JREs). Although I guess it would still allow us to ignore remote DTDs in the event a non-standard SAX-based parser was used (that does not support the load-external-dtd feature.
Catching the exception definitely makes sense.

But I wouldn't log it. It's just a bug in Xerces that we have to set this property at all. In all other places in the SDK where we set the feature via org.xml.sax.XMLReader#setFeature(..), we also silently catch the exception.
updated fix: http://git.eclipse.org/c/jdt/eclipse.jdt.debug.git/commit/?id=b3950a3f7f750f6c073b0e94012b1ea36b1719e1
