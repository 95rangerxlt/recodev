> Is that a bug though, should we wait for the frame to load before calling

Yeah, that's a good point - it's a bit ambiguous where this should be handled.

We definitely don't want to skip setting menuEnabled = true from webide.js even if the frame isn't loaded (see udateProjectEditorMenusVisibility), since this value matters once the menus are opened.  I can think of two alternative ways to handle it:

1) Manage this from the web ide like so:

this.projecteditor.loaded.then(() => { this.projecteditor.menuEnabled = shouldBeEnabled; }, console.error)

But I believe this will still spam the logs with the errors when/if the loaded promise is rejected.

2) Or, we could not ever call _updateMenuItems from the menuEnabled setter - which would probably be OK since that gets called on popupshown for the menus.  This would be more consistent than special casing the !this._loaded situation.  The only time that would be an issue is if a menu item is requested to be disabled while the menu was open to start with (probably not very likely).

IMO either the proposed way in this patch or alternative 2 would be ideal as far as avoiding logspam.
