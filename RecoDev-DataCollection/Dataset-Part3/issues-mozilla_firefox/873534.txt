Most, if not all of those debugger tests use a function called debug_tab_pane to set things up, which is located in head.js:

https://mxr.mozilla.org/mozilla-central/source/browser/devtools/debugger/test/head.js#168

That function, in turn, uses addTab to open some content in a new tab. addTab is defined here: https://mxr.mozilla.org/mozilla-central/source/browser/devtools/debugger/test/head.js#65

I'm able to reproduce this just by attempting to run the debugger tests on a debug build with ./mach mochitest-browser browser/devtools/debugger/test.

I did a little digging, and here's what it looks like:

Most, if not all of those debugger tests use a function called debug_tab_pane to set things up, which is located in head.js:

https://mxr.mozilla.org/mozilla-central/source/browser/devtools/debugger/test/head.js#168

That function, in turn, uses addTab to open some content in a new tab. addTab is defined here: https://mxr.mozilla.org/mozilla-central/source/browser/devtools/debugger/test/head.js#65

It looks like the tests hang because the ready state of the page never reaches "complete" or "interactive". From some printf debugging, it looks like the page hits "loading" and doesn't move from there. If I hit the refresh button while the test is waiting, it seems to do the job of kicking it forward.

So, not sure why the complete/interactive state isn't being set. But that's where I got to with it. Care to pick up the ball, Gijs?
